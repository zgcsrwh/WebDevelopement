# WebDevelopment 最终验收清单

本清单基于你当前这版 React 单页程序，以及项目正在使用的真实 Firestore 集合结构整理：

- `member`
- `admin_staff`
- `facility`
- `request`
- `repair`
- `profile`
- `matching`
- `friends`
- `notification`
- `time_slot`

建议在最终提交前，严格按照本清单完成一次验收。

## 1. 测试前的重要说明

### 1.1 只读检查
下面这些操作优先做，通常不会创建新的业务数据：

- 登录
- 打开首页、列表页、详情页
- 查看场馆信息
- 查看我的预约
- 查看通知
- 查看员工工作台列表
- 查看管理员列表页
- 使用筛选、搜索、分页式浏览

### 1.2 会修改真实数据库的操作
下面这些操作会直接写入 Firestore，必须谨慎测试：

- 会员注册
- 修改个人资料
- 提交预约
- 撤回预约
- 取消已通过预约
- 提交报修
- 创建或修改匹配档案
- 发送、接受、拒绝匹配请求
- 通知标记已读
- 员工审批预约
- 员工核销到场
- 员工解决报修
- 管理员新增员工
- 管理员编辑员工
- 管理员停用员工
- 管理员新增场馆
- 管理员编辑场馆
- 管理员删除场馆
- 删除账号

### 1.3 推荐测试方式

- 准备专门的 `Member`、`Staff`、`Admin` 测试账号。
- 所有测试写入数据都加明显前缀，比如 `TEST booking`、`TEST repair`、`TEST match`。
- 预约测试尽量统一用同一个未来日期，方便后续清理。
- 大量写操作前，建议先把数据库再导出备份一次。
- 先做只读检查，再做写入检查。

## 2. Member 角色验收

### 2.1 登录与路由拦截

- 使用会员账号登录。
- 预期结果：
- 成功进入 `/home`
- 导航栏显示 `Overview`、`Facilities`、`My Bookings`、`Reports`、`Partner`、`Profile`
- 强行访问 `/staff/...` 或 `/admin/...` 会被重定向

### 2.2 首页工作台

- 打开 `/home`
- 预期结果：
- Upcoming bookings 数量来自 `request`
- Pending requests 数量来自 `request`
- Open repair reports 数量来自 `repair`
- Pending partner requests 数量来自 `matching`
- Recent notifications 来自 `notification`

### 2.3 场馆浏览与跳转

- 打开 `/facilities`
- 切换日期、类型、状态、时间段筛选
- 打开某个场馆详情页
- 预期结果：
- 只显示当前可见场馆
- `normal` 状态场馆可进入预约
- 日期只能选今天到未来 7 天
- 场馆详情页与返回列表跳转正常

### 2.4 提交预约

- 从 `/bookings/new` 直接进入
- 从场馆详情页进入预约
- 预期结果：
- 从场馆详情页进入时，场馆会自动带入
- 只能选择开放时段
- 时长限制为 1 到 4 小时
- 人数不能超过场馆容量
- 邀请好友数量不能超过 `attendees - 1`
- 被邀请的人必须来自真实 `friends` 集合中的好友

- 写入测试：
- 提交一条带 `TEST booking` 描述的预约
- 预期结果：
- `request` 中生成一条 `pending` 记录
- 对应 `time_slot` 被锁定
- 发起人和被邀请好友收到通知
- 负责该场馆的员工收到通知

### 2.5 我的预约

- 打开 `/bookings`
- 预期结果：
- 能看到自己发起的预约
- 也能看到自己作为参与人的预约
- Pending、Upcoming、History 分类正常
- `/bookings/:id` 详情页可以正常进入

- 写入测试：
- 撤回一条 `pending` 或 `suggested` 预约
- 预期结果：
- 该预约变为 `cancelled`
- 对应锁定时段恢复为 `open`

- 写入测试：
- 取消一条距离开始时间至少 2 小时以上的 `accepted` 预约
- 预期结果：
- 该预约变为 `cancelled`
- 会员 `cancel_times` 增加
- 时段释放

### 2.6 报修

- 打开 `/reports`
- 预期结果：
- 只能看到当前会员自己的报修记录
- 描述长度限制为 500 字以内

- 写入测试：
- 提交一条 `TEST repair`
- 预期结果：
- `repair` 新增一条 `pending`
- 对应场馆状态表现为 `fixing`
- 会员和场馆负责员工收到通知

### 2.7 伙伴匹配

- 打开 `/partner`
- 预期结果：
- 可以创建或更新自己的匹配档案
- 可以开启或关闭匹配

- 写入测试：
- 保存一份完整匹配档案
- 预期结果：
- `profile` 中为当前会员创建或更新唯一档案
- 重复保存是更新，不是重复新增

- 写入测试：
- 关闭匹配
- 预期结果：
- `open_match` 变为 `false`
- 相关待处理匹配请求失效

- 打开 `/partner/discover`
- 预期结果：
- 如果当前会员还没开启匹配，会被拦截并提示先去完善档案
- 如果已开启匹配，只能看到其他开启匹配且账号有效的会员

- 打开 `/partner/requests`
- 预期结果：
- 收到的请求与发出的请求分开展示

- 写入测试：
- 发送一条 `TEST match`
- 接受一条请求
- 拒绝一条请求
- 预期结果：
- `matching` 中生成或更新对应记录
- 接受后双方在 `friends` 中互相建立好友关系
- 拒绝后请求状态变更为 `rejected`

### 2.8 个人资料与账号删除

- 打开 `/profile`
- 预期结果：
- 姓名、生日、地址可编辑
- 好友列表来自真实 `friends`
- 删除资格检查会读取真实预约和报修状态

- 写入测试：
- 在存在未完成预约或未解决报修时执行删除资格检查
- 预期结果：
- 系统明确阻止删除，并返回具体原因

## 3. Staff 角色验收

### 3.1 登录与路由拦截

- 使用员工账号登录
- 预期结果：
- 成功进入 `/staff/requests`
- 导航显示 `Requests`、`Check-in`、`Reports`、`Profile`
- 访问会员或管理员专属路由会被重定向

### 3.2 预约审批

- 打开 `/staff/requests`
- 预期结果：
- 只能看到分配给自己负责场馆的待审批预约
- 搜索和日期筛选可用

- 写入测试：
- 通过一条预约
- 拒绝一条预约并填写反馈
- 建议修改一条预约并填写反馈
- 预期结果：
- 通过后状态变为 `accepted`
- 拒绝后状态变为 `rejected`，并保存 `staff_response`
- 建议后状态变为 `suggested`，并保存 `staff_response`
- 拒绝和建议会释放锁定时段
- 发起人和参与人收到通知

### 3.3 到场核销

- 打开 `/staff/bookings`
- 预期结果：
- 只能看到分配给自己的 `accepted` 或 `in_progress` 预约

- 写入测试：
- 在允许的核销时间窗口内核销一条预约
- 预期结果：
- 预约状态变为 `in_progress`
- 参与人收到通知

### 3.4 报修处理

- 打开 `/staff/reports`
- 预期结果：
- 只能看到自己负责场馆对应的报修单
- 搜索和状态筛选可用

- 写入测试：
- 将一条 `pending` 报修标记为 `resolved`
- 预期结果：
- `repair.status` 更新为 `resolved`
- `completed_at` 被写入
- 报修会员收到通知
- 如果该场馆已没有其他 `pending` 报修，则场馆不再表现为 `fixing`

### 3.5 员工资料页

- 打开 `/staff/profile`
- 预期结果：
- 账号信息显示正常
- 自己负责的场馆显示正常
- Recent workload 来自真实预约与报修数据

## 4. Admin 角色验收

### 4.1 登录与路由拦截

- 使用管理员账号登录
- 预期结果：
- 成功进入 `/admin/facilities`
- 导航显示 `Facilities` 和 `Staff`
- 访问不属于管理员的工作台路由会被重定向

### 4.2 场馆管理

- 打开 `/admin/facilities`
- 预期结果：
- 场馆列表、状态、负责员工、营业时间和筛选功能正常
- 容量输入支持 1 到 200
- 开放时间和关闭时间校验正常

- 写入测试：
- 新增一个 `TEST facility`
- 预期结果：
- `facility` 新增记录
- 后续可预约日期的 `time_slot` 会准备好

- 写入测试：
- 编辑一个已有场馆
- 预期结果：
- 名称、类型、描述、使用指南、容量、时间、地点、状态、`staff_id` 都可更新

- 写入测试：
- 删除一个场馆
- 预期结果：
- 场馆状态改为 `deleted`
- 相关待处理或已通过预约会被拒绝

### 4.3 员工管理

- 写入测试：

- 预期结果：
- Firebase Auth 中创建新账号
- `admin_staff` 中新增 `staff` 角色且状态为 `active`

- 写入测试：
- 编辑一个已有员工
- 预期结果：
- 姓名、生日、地址可更新

- 写入测试：
- 停用一名员工
- 预期结果：
- 员工状态变为 `inactive`
- 该员工负责的场馆清空 `staff_id`

- 相关待审批或已通过预约被拒绝并释放时段

## 6. 如何尽量不把真实数据库弄乱

如果你想检查功能是否完整，但又尽量不污染真实库，推荐这样做：

- 第一步：三个角色分别登录，只做只读检查
- 第二步：确认页面、跳转、展示、权限都没问题后，再开始写入测试
- 第三步：写入测试只使用专门测试账号
- 第四步：所有测试数据统一加 `TEST` 前缀
- 第五步：测试结束后按 `TEST` 前缀集中清理

## 7. 可以认为接近提交状态的判断标准

满足下面这些条件时，基本可以认为项目已经达到提交要求：

- 三个角色都能正常登录
- 路由跳转和权限拦截正确
- 会员预约、报修、匹配、个人资料功能完整
- 员工审批、核销、报修处理完整
- 管理员场馆管理、员工管理完整
- 通知联动完整
- `eslint src` 为 `0 error / 0 warning`
- 生产构建成功
